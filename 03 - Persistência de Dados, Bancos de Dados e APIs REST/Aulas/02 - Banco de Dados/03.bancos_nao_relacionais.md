# Bancos Não Relacionais

Os Bancos Não Relacionais foram criados para oferecer flexibilidade e velocidade ao lidar com volumes massivos de dados, muitas vezes desstruturados.

Em vez de forçar os dados a se encaixarem em linhas e colunas pré-definidas, o NoSQL permite armazenar informações de diversas formas. Os quatro tipos principais são:

1. **Documentos:** Armazenam dados em formatos semelhantes ao JSON.
2. **Chave-Valor:** O modelo mais simples, como um dicionário.
3. **Grafos:** Focado em conexões e redes complexas.
4. **Colunares:** Otimizados para leitura rápida de grandes volumes de dados.

A grande vantagem desse modelo é a facilidade de escalar o banco de dados (distribuir em vários servidores) e a liberdade para alterar a estrutura dos dados a qualquer momento, sem quebrar o sistema.

## Chave Primária

Nos bancos não relacionais, o conceito de chave primária ainda existe, mas costuma ser chamado simplesmente de ID, Document ID ou Key.

Assim como no modelo relacional, é essencial ter um identificador único para encontrar um registro específico.

## Chave Estrangeira

Bancos não relacionais não possuem chaves estrangeiras tradicionais. Eles não forçam a integridade referencial. A responsabilidade de manter tudo organizado passa a ser do código da aplicação.

Para lidar com dados que pertecem uns aos outros, o NoSQL usa duas estratégias principais:

1. **Aninhamento (Embedding):** Em vez de separar os dados em tabelas diferentes e conectá-los com uma chave estrangeira, você guarda tudo no mesmo lugar.
2. **Referenciamento (Referencing):** Quando os dados são muito grandes para serem aninhados, você salva o ID de um documento dentro de outro. Funciona como uma chave estrangeira, mas o bando de dados não faz a junção (JOIN) automaticamente para você. Sua aplicação precisa fazer duas buscas separadas.
